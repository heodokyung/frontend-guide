## 현재 기준

2026년 5월 기준으로 실무에서 함께 봐야 할 기준은 두 가지입니다.

| 기준 | 설명 |
|---|---|
| WCAG 2.2 | W3C의 웹 콘텐츠 접근성 국제 표준. 2023년 10월 5일 W3C Recommendation으로 공개되었고, 2024년 12월 12일 편집·정의 업데이트가 있었습니다. |
| KWCAG 2.2 | 한국형 웹 콘텐츠 접근성 지침 2.2. 국내 웹 접근성 품질인증과 공공/기업 사이트 점검에서 자주 기준으로 사용됩니다. |

WCAG 2.2는 WCAG 2.1을 대체한다기보다 확장합니다. 기존 2.0/2.1의 성공 기준은 대부분 유지되고, WCAG 2.2에서 새 성공 기준이 추가되었습니다. 단, `4.1.1 Parsing`은 WCAG 2.2에서 obsolete 처리되어 제거되었습니다.

실무에서는 아래처럼 잡으면 안전합니다.

```txt
국제 기준: WCAG 2.2 AA를 목표로 본다.
국내 기준: KWCAG 2.2 검사 항목을 함께 확인한다.
```

## 접근성을 보는 관점

접근성은 특정 사용자만을 위한 배려가 아닙니다. 키보드만 쓰는 사람, 화면을 확대해서 보는 사람, 스크린리더를 쓰는 사람, 색 구분이 어려운 사람, 손이 불편한 사람, 모바일에서 흔들리는 환경에 있는 사람까지 모두 포함합니다.

접근성은 아래 네 원칙으로 나눠 생각하면 쉽습니다.

| 원칙 | 질문 |
|---|---|
| 인식 가능 | 사용자가 콘텐츠를 볼 수 있거나 들을 수 있는가? |
| 운용 가능 | 키보드, 포인터, 터치 등으로 조작할 수 있는가? |
| 이해 가능 | 화면과 입력 흐름이 예측 가능한가? |
| 견고함 | 브라우저와 보조기술이 구조를 이해할 수 있는가? |

## WCAG 2.2에서 특히 봐야 할 변화

WCAG 2.2는 2.1 이후 9개의 성공 기준을 추가했습니다.

| 번호 | 이름 | 레벨 | 프론트엔드에서 볼 것 |
|---|---|---:|---|
| 2.4.11 | Focus Appearance | AA | 포커스 표시가 충분히 보이는가 |
| 2.4.12 | Focus Not Obscured (Minimum) | AA | sticky header, toast, modal 때문에 포커스가 가려지지 않는가 |
| 2.4.13 | Focus Not Obscured (Enhanced) | AAA | 포커스가 완전히 가려지지 않는가 |
| 2.5.7 | Dragging Movements | AA | 드래그 말고 클릭/버튼으로도 같은 기능을 쓸 수 있는가 |
| 2.5.8 | Target Size (Minimum) | AA | 작은 버튼, 아이콘 버튼의 클릭 영역이 충분한가 |
| 3.2.6 | Consistent Help | A | 도움말/문의/지원 경로가 페이지마다 일관된가 |
| 3.3.7 | Redundant Entry | A | 사용자가 이미 입력한 정보를 반복해서 요구하지 않는가 |
| 3.3.8 | Accessible Authentication (Minimum) | AA | 기억력 테스트, 퍼즐, 복잡한 인증에만 의존하지 않는가 |
| 3.3.9 | Accessible Authentication (Enhanced) | AAA | 더 강한 수준의 접근 가능한 인증을 제공하는가 |

추가 기준 중 프론트엔드 작업에서 특히 자주 걸리는 항목은 `Focus Appearance`, `Focus Not Obscured`, `Dragging Movements`, `Target Size`, `Redundant Entry`, `Accessible Authentication`입니다.

## KWCAG 2.2 실무 점검 항목

국내 기준은 KWCAG 2.2의 4대 원칙과 33개 검사 항목을 기본으로 봅니다. 아래는 프론트엔드 작업자가 코드 리뷰 때 바로 볼 수 있도록 재정리한 목록입니다.

### 1. 인식의 용이성

- [ ] 텍스트 아닌 콘텐츠에 대체 텍스트를 제공했는가?
- [ ] 멀티미디어 콘텐츠에 자막, 원고, 수화 등 대체 수단이 있는가?
- [ ] 표가 이해하기 쉬운 구조로 작성되었는가?
- [ ] 콘텐츠가 논리적인 순서로 제공되는가?
- [ ] 지시사항이 색, 모양, 위치, 방향에만 의존하지 않는가?
- [ ] 색상과 관계없이 정보를 이해할 수 있는가?
- [ ] 자동 재생되는 소리가 없는가?
- [ ] 텍스트와 배경의 명도 대비가 충분한가?
- [ ] 이웃한 콘텐츠가 시각적으로 구분되는가?

### 2. 운용의 용이성

- [ ] 모든 기능을 키보드로 사용할 수 있는가?
- [ ] 키보드 초점 이동 순서가 논리적인가?
- [ ] 초점 표시가 명확하게 보이는가?
- [ ] 입력과 컨트롤의 조작 영역이 충분한가?
- [ ] 문자 단축키가 오동작을 만들지 않는가?
- [ ] 시간 제한이 있다면 연장/해제할 수 있는가?
- [ ] 자동으로 움직이는 콘텐츠를 정지할 수 있는가?
- [ ] 깜빡임과 번쩍임이 위험 수준이 아닌가?
- [ ] 반복 영역을 건너뛸 수 있는가?
- [ ] 페이지, 프레임, 콘텐츠 블록에 제목이 있는가?
- [ ] 현재 위치와 참조 위치를 알 수 있는가?
- [ ] 드래그, 핀치, 흔들기 같은 동작 기반 기능에 대체 수단이 있는가?

### 3. 이해의 용이성

- [ ] 문서의 기본 언어가 표시되어 있는가?
- [ ] 사용자가 의도하지 않은 새 창, 초점 변화, 자동 실행이 없는가?
- [ ] 도움 정보가 일관된 위치와 순서에 있는가?
- [ ] 입력 오류를 찾고 고칠 수 있는가?
- [ ] 입력 요소에 레이블이 있는가?
- [ ] 인증이 인지 기능 테스트에만 의존하지 않는가?
- [ ] 같은 정보를 반복해서 입력하게 만들지 않는가?

### 4. 견고성

- [ ] 마크업 오류가 보조기술 해석을 방해하지 않는가?
- [ ] UI 컴포넌트의 이름, 역할, 상태를 프로그램적으로 알 수 있는가?
- [ ] 동적으로 바뀐 상태가 스크린리더에 전달되는가?

## 프론트엔드 구현 체크리스트

### HTML

```html
<!-- Good -->
<button type="button">저장</button>
<a href="/projects">프로젝트 보기</a>
```

- [ ] 버튼과 링크를 구분했는가?
- [ ] 제목 계층이 `h1 → h2 → h3` 순서로 자연스러운가?
- [ ] 이미지 `alt`를 의미에 맞게 작성했는가?
- [ ] 장식 이미지는 `alt=""`로 처리했는가?
- [ ] 폼 입력과 `label`이 연결되어 있는가?

### CSS

```css
.button:focus-visible {
  outline: 3px solid var(--color-focus);
  outline-offset: 2px;
}
```

- [ ] `:focus-visible` 스타일이 있는가?
- [ ] 텍스트 대비가 충분한가?
- [ ] 작은 아이콘 버튼의 터치 영역이 충분한가?
- [ ] `prefers-reduced-motion`을 고려했는가?
- [ ] 화면 확대 시 콘텐츠가 겹치지 않는가?

### JavaScript

```js
const dialog = document.querySelector('[data-dialog]')
const openButton = document.querySelector('[data-dialog-open]')
const closeButton = document.querySelector('[data-dialog-close]')

openButton.addEventListener('click', () => {
  dialog.showModal()
  closeButton.focus()
})
```

- [ ] 동적으로 열린 UI에 초점이 이동하는가?
- [ ] 닫을 때 초점이 원래 위치로 돌아오는가?
- [ ] 상태 변화가 필요한 경우 `aria-expanded`, `aria-selected`, `aria-current`를 갱신하는가?
- [ ] 에러 메시지는 `aria-describedby` 또는 `role="alert"`로 연결되어 있는가?

## 컴포넌트별 점검 기준

### 버튼

- `button` 요소를 사용합니다.
- 아이콘만 있는 버튼에는 접근 가능한 이름을 제공합니다.
- 비활성 상태는 `disabled` 또는 `aria-disabled`를 의도에 맞게 사용합니다.

```html
<button type="button" aria-label="검색 열기">
  <svg aria-hidden="true" focusable="false"></svg>
</button>
```

### 링크

- 목적이 드러나는 링크 텍스트를 사용합니다.
- “여기”, “자세히 보기”만 반복하지 않습니다.

```html
<!-- Bad -->
<a href="/guide">여기</a>

<!-- Good -->
<a href="/guide">프론트엔드 가이드 보기</a>
```

### 폼

- 모든 입력에는 레이블을 연결합니다.
- 필수 여부와 오류 메시지를 시각적으로만 전달하지 않습니다.
- 오류가 나면 무엇을 어떻게 고쳐야 하는지 알려줍니다.

```html
<label for="password">비밀번호</label>
<input id="password" type="password" aria-describedby="password-error">
<p id="password-error">비밀번호는 8자 이상 입력해주세요.</p>
```

### 모달

- 열릴 때 모달 내부로 초점이 이동해야 합니다.
- 닫을 때 원래 버튼으로 초점이 돌아와야 합니다.
- `Esc`로 닫을 수 있어야 합니다.
- 배경 스크롤 잠금과 포커스 트랩을 확인합니다.

### 탭

- `role="tablist"`, `role="tab"`, `role="tabpanel"`을 사용할 수 있습니다.
- 선택 상태는 `aria-selected`로 전달합니다.
- 키보드 방향키 이동을 지원할지 컴포넌트 기준을 정합니다.

### 토스트 / 알림

- 단순 성공 메시지는 `role="status"`를 사용합니다.
- 즉시 알아야 하는 오류는 `role="alert"`를 사용할 수 있습니다.
- 너무 빨리 사라져서 읽을 수 없는 메시지는 피합니다.

## 테스트 방법

자동 검사만으로 접근성을 보장할 수는 없습니다. 아래 순서로 같이 봅니다.

1. 키보드만으로 전체 화면을 사용해 봅니다.
2. 브라우저 확대 200%에서 레이아웃을 확인합니다.
3. 색 대비 검사 도구로 텍스트 대비를 봅니다.
4. Lighthouse, axe DevTools 같은 자동 검사 도구를 돌립니다.
5. 스크린리더로 주요 흐름을 들어봅니다.
6. 실제 입력 오류, 모달 열림/닫힘, 탭 이동 같은 동적 상태를 확인합니다.

빠른 키보드 점검:

```txt
Tab          다음 초점
Shift + Tab  이전 초점
Enter        버튼/링크 실행
Space        버튼 실행, 체크박스 토글
Esc          모달/팝업 닫기
Arrow keys   탭/라디오/슬라이더 등에서 사용 가능
```

## 자주 놓치는 부분

- 커스텀 셀렉트가 키보드로 열리지 않음
- 모달이 열렸는데 초점은 뒤 배경에 남아 있음
- 닫기 버튼이 아이콘만 있고 이름이 없음
- 포커스 스타일을 CSS reset에서 지워버림
- `div`에 클릭 이벤트만 붙이고 키보드 처리를 하지 않음
- 입력 오류 메시지가 시각적으로만 보임
- 색상만으로 성공/오류를 구분함
- sticky header가 포커스된 요소를 가림
- 드래그로만 순서를 바꿀 수 있고 버튼 대체 수단이 없음
- 캡차나 기억 테스트가 인증의 유일한 방법임
