# Markdown Guide

> README, 문서, 이슈, PR 설명을 읽기 쉽게 작성하기 위한 Markdown 실무 가이드입니다.

Markdown은 단순히 글자를 꾸미는 문법이 아닙니다. 좋은 Markdown 문서는 읽는 사람이 빠르게 구조를 파악하고, 필요한 정보를 바로 찾고, 예시를 복사해서 적용할 수 있게 만듭니다.

---

## 먼저 결론

좋은 Markdown 문서는 아래 네 가지를 지킵니다.

1. 제목 구조가 명확하다.
2. 문단이 너무 길지 않다.
3. 코드와 설명이 분리되어 있다.
4. 표, 목록, 체크리스트는 “정리”가 필요할 때만 사용한다.

---

## 기본 문법 빠른 예시

```md
# 문서 제목

> 문서가 해결하는 문제를 한 문장으로 설명합니다.

## 섹션 제목

본문을 작성합니다.

- 목록 1
- 목록 2

1. 첫 번째
2. 두 번째

**강조**와 _기울임_을 사용할 수 있습니다.

[링크 텍스트](https://example.com)

![대체 텍스트](./image.png)

`inline code`

```js
console.log('code block')
```
```

## 제목 작성 규칙

제목은 문서의 구조입니다. 글자 크기를 키우기 위한 장식으로 쓰지 않습니다.

```md
# 문서 제목
## 큰 섹션
### 세부 섹션
#### 보조 설명
```

권장 규칙:

- 한 문서에는 `#` 제목을 하나만 사용합니다.
- 제목 단계는 건너뛰지 않습니다.
- 제목은 짧고 명확하게 작성합니다.
- 이모지는 꼭 필요한 경우에만 사용합니다.

```md
<!-- Bad -->
# Git
### Push 오류

<!-- Good -->
# Git Guide
## 문제 해결
### Push 오류
```

## 문단과 줄바꿈

Markdown에서 줄바꿈은 렌더러에 따라 다르게 보일 수 있습니다. 문단을 나누고 싶다면 빈 줄을 넣습니다.

```md
첫 번째 문단입니다.

두 번째 문단입니다.
```

강제 줄바꿈은 문장 끝에 공백 두 칸을 넣는 방식도 있지만, 문서 유지보수 관점에서는 `<br>`이나 문단 분리를 더 명확히 사용하는 편이 낫습니다.

---

## 강조

```md
**중요한 내용**
_보조 강조_
~~취소선~~
```

권장:

- 핵심 키워드는 `**굵게**` 처리합니다.
- 긴 문장 전체를 굵게 만들지 않습니다.
- 강조가 많으면 강조가 사라집니다.

## 목록

### 순서 없는 목록

```md
- Git 상태 확인
- 변경 파일 스테이징
- 커밋 작성
- 원격 저장소로 Push
```

### 순서 있는 목록

```md
1. GitHub에서 레포지토리를 만든다.
2. 로컬 폴더에서 remote를 연결한다.
3. 첫 커밋을 push한다.
```

권장:

- 절차는 숫자 목록을 사용합니다.
- 병렬 개념은 불릿 목록을 사용합니다.
- 목록 안에 목록을 너무 깊게 중첩하지 않습니다.

---

## 체크리스트

GitHub에서는 체크리스트를 사용할 수 있습니다.

```md
- [x] 레포지토리 생성
- [x] README 작성
- [ ] GitHub Pages 설정
- [ ] 링크 확인
```

체크리스트는 작업 상태를 보여줄 때 유용합니다. 단순 설명 문서에는 남용하지 않습니다.

---

## 인용문

```md
> 핵심 메시지나 주의사항을 강조할 때 사용합니다.
```

인용문은 아래 경우에 적합합니다.

- 문서의 목적 설명
- 중요한 결론
- 경고나 주의사항

## 코드 작성

### 인라인 코드

명령어, 파일명, 변수명은 인라인 코드로 표시합니다.

```md
`git status`를 먼저 실행합니다.
`README.md` 파일을 수정합니다.
```

### 코드 블록

코드 블록에는 가능하면 언어를 지정합니다.

```md
```bash
git status
git add .
git commit -m "docs: update README"
```
```

```md
```html
<button type="button">저장</button>
```
```

권장:

- 명령어 예시는 `bash`를 사용합니다.
- HTML 예시는 `html`, CSS 예시는 `css`, JS는 `js`, TS는 `ts`를 사용합니다.
- 코드 블록 안에는 불필요한 프롬프트 기호 `$`를 넣지 않는 편이 복사하기 쉽습니다.

## 링크

```md
[GitHub Docs](https://docs.github.com)
```

권장:

- “여기 클릭”보다 목적을 드러내는 텍스트를 사용합니다.
- 외부 링크는 신뢰 가능한 공식 문서를 우선합니다.
- 내부 문서는 상대 경로를 사용합니다.

```md
<!-- Bad -->
[여기](./docs/git/README.md)를 클릭하세요.

<!-- Good -->
[Git 가이드](./docs/git/README.md)를 확인하세요.
```

## 표

표는 비교할 항목이 있을 때만 사용합니다.

```md
| 상황 | 확인할 명령어 | 다음 행동 |
|---|---|---|
| remote 확인 | `git remote -v` | 주소가 맞는지 확인 |
| 변경 파일 확인 | `git status` | 필요한 파일만 stage |
| 커밋 확인 | `git log --oneline -5` | 최근 커밋 확인 |
```

표가 너무 길어지면 가독성이 떨어집니다. 설명이 길다면 섹션을 나눕니다.

## 접기/펼치기

GitHub Markdown에서는 HTML `details` 태그를 사용할 수 있습니다.

```md
<details>
<summary>자세히 보기</summary>

숨겨진 설명입니다.

</details>
```

긴 로그, 긴 예시, 부가 설명을 접을 때 유용합니다.

## README 기본 구조

레포지토리 루트의 README는 아래 순서를 추천합니다.

```md
# 프로젝트 이름

> 프로젝트를 한 문장으로 설명합니다.

## 소개

무엇을 위한 프로젝트인지 설명합니다.

## 주요 기능

- 기능 1
- 기능 2

## 기술 스택

- Vue
- TypeScript
- Vite

## 시작하기

```bash
npm install
npm run dev
```

```txt
src/
├─ components/
├─ views/
└─ stores/
```

## 문서

- [Git 가이드](./docs/git/README.md)

## 라이선스

MIT
```

## 문서형 README 구조

가이드 문서라면 아래 구조가 좋습니다.

```md
# 문서 제목

> 이 문서가 해결하는 문제를 한 문장으로 설명합니다.

## 먼저 결론

가장 중요한 판단 기준을 먼저 적습니다.

## 언제 사용하는가

이 문서가 필요한 상황을 적습니다.

## 기본 규칙

반드시 지킬 기준을 적습니다.

## 예시

Bad / Good 예시를 함께 제공합니다.

## 체크리스트

실제 적용 전 확인할 항목을 정리합니다.
```

## 좋은 Markdown 체크리스트

문서를 올리기 전 아래를 확인합니다.

- [ ] 제목 단계가 자연스럽다.
- [ ] 한 문단이 너무 길지 않다.
- [ ] 코드 블록에 언어가 지정되어 있다.
- [ ] 링크 텍스트만 봐도 목적을 알 수 있다.
- [ ] 이미지에 의미 있는 대체 텍스트가 있다.
- [ ] 표가 너무 넓거나 길지 않다.
- [ ] 체크리스트는 실제 작업 상태에만 사용했다.
- [ ] README 첫 화면에서 프로젝트 목적이 바로 보인다.

## 자주 하는 실수

### 제목을 글자 크기 조절용으로 쓰는 경우

```md
<!-- Bad -->
# 큰 글씨
#### 작은 글씨
```

제목은 문서 구조를 표현하기 위해 사용합니다.

### 코드 블록 언어를 생략하는 경우

````md
<!-- Bad -->
```
const name = 'guide'
```

<!-- Good -->
```js
const name = 'guide'
```
````

### 링크 텍스트가 모호한 경우

```md
<!-- Bad -->
[클릭](https://docs.github.com)

<!-- Good -->
[GitHub Markdown 문서](https://docs.github.com)
```
