# Markdown Guide

> README, 문서, 이슈, PR 설명을 읽기 쉽게 작성하기 위한 Markdown 작성 기준입니다.

Markdown은 문법을 많이 아는 것보다 구조를 잘 잡는 것이 더 중요합니다. 읽는 사람이 첫 화면에서 목적을 이해하고, 필요한 부분으로 바로 이동할 수 있으면 좋은 문서입니다.

## 기본 원칙

좋은 Markdown 문서는 아래 기준을 지킵니다.

1. 제목만 읽어도 흐름이 보입니다.
2. 한 문단이 너무 길지 않습니다.
3. 코드와 설명이 분리되어 있습니다.
4. 표는 비교가 필요할 때만 씁니다.
5. 목록은 정리가 필요할 때만 씁니다.
6. 링크 텍스트만 봐도 어디로 가는지 알 수 있습니다.
7. 예시는 복사해서 바로 쓸 수 있어야 합니다.

## 제목 작성

제목은 글자 크기를 키우기 위한 장식이 아니라 문서의 구조입니다.

```md
# 문서 제목
## 큰 섹션
### 세부 섹션
#### 보조 설명
```

권장 기준:

- 한 문서에는 `#` 제목을 하나만 사용합니다.
- 제목 단계는 건너뛰지 않습니다.
- 제목은 짧고 명확하게 씁니다.
- 이모지는 제목을 구분할 때만 적게 사용합니다.

```md
<!-- Bad -->
# Git
### Push 오류

<!-- Good -->
# Git Guide
## 문제 해결
### Push 오류
```

## 문단과 줄바꿈

문단을 나누고 싶다면 빈 줄을 넣습니다.

```md
첫 번째 문단입니다.

두 번째 문단입니다.
```

줄 끝 공백 두 칸으로 강제 줄바꿈을 만들 수도 있지만, 유지보수 관점에서는 문단을 나누거나 `<br>`을 명시하는 편이 더 분명합니다.

문장이 길어질 때는 한 문단에 너무 많은 이야기를 넣지 않습니다. GitHub README는 모바일에서도 자주 읽히기 때문에 짧은 문단이 유리합니다.

---

## 목록

절차는 숫자 목록을 사용합니다.

```md
1. GitHub에서 레포지토리를 만든다.
2. 로컬 폴더에서 remote를 연결한다.
3. 첫 커밋을 push한다.
```

병렬 개념은 불릿 목록을 사용합니다.

```md
- Git
- Markdown
- Markup
- CSS
```

목록 안에 목록을 넣을 수 있지만, 3단계 이상 깊어지면 표나 별도 섹션으로 분리하는 것이 좋습니다.

## 표

표는 비교와 매핑에 좋습니다.

```md
| 상황 | 볼 문서 |
|---|---|
| GitHub 연결이 필요하다 | Git 입문 가이드 |
| README를 다듬고 싶다 | Markdown 가이드 |
```

권장 기준:

- 표 안의 문장은 짧게 씁니다.
- 긴 설명은 표 아래 문단으로 뺍니다.
- 열이 4개 이상이면 읽기 어려울 수 있습니다.
- 모바일에서 표가 좁아지는지 확인합니다.

## 코드

### 인라인 코드

명령어, 파일명, 변수명은 인라인 코드로 표시합니다.

```md
`git status`를 먼저 실행합니다.
`README.md` 파일을 수정합니다.
`isActive` 값으로 상태를 판단합니다.
```

### 코드 블록

코드 블록에는 언어를 지정합니다.

````md
```bash
git status
git add .
git commit -m "docs: update README"
```
````

자주 쓰는 언어명:

| 언어 | 코드블록 |
|---|---|
| Shell | `bash` |
| HTML | `html` |
| CSS | `css` |
| JavaScript | `js` |
| TypeScript | `ts` |
| JSON | `json` |
| Markdown | `md` |

명령어 예시에 `$`를 넣지 않으면 복사하기 쉽습니다.

```bash
# Good
git status

# 복사할 때 $까지 복사될 수 있음
$ git status
```

## 링크와 이미지

링크 텍스트는 목적을 드러내야 합니다.

```md
<!-- Bad -->
[여기](./docs/git/README.md)를 클릭하세요.

<!-- Good -->
[Git 가이드](./docs/git/README.md)를 확인하세요.
```

내부 문서는 상대 경로를 사용합니다.

```md
[Markup 가이드](../markup/README.md)
```

이미지에는 대체 텍스트를 작성합니다.

```md
![Frontend Guide 배너](./assets/banner.png)
```

장식 이미지를 Markdown에서 완전히 장식으로 처리하기는 어렵습니다. 중요한 정보가 없는 이미지는 본문 설명을 따로 제공하지 않는 방식으로 부담을 줄입니다.

## 체크리스트

GitHub에서는 체크리스트를 사용할 수 있습니다.

```md
- [x] 레포지토리 생성
- [x] README 작성
- [ ] GitHub Pages 설정
- [ ] 링크 확인
```

체크리스트는 작업 상태를 보여줄 때 좋습니다. 설명 문서 전체를 체크리스트로 만들면 읽기 피곤해질 수 있으니, 리뷰나 배포 전 점검에 사용합니다.

## 접기/펼치기

긴 예시나 부가 설명은 `<details>`를 사용할 수 있습니다.

````md
<details>
<summary>자세한 명령어 보기</summary>

```bash
git status
git add .
git commit -m "docs: update"
git push
```

</details>
````

주의할 점:

- `<summary>` 아래에는 빈 줄을 넣습니다.
- 코드블록을 넣을 때 열고 닫는 백틱 개수를 조심합니다.
- 꼭 숨길 필요가 있는 긴 내용에만 씁니다.

## 강조와 인용

```md
**중요한 내용**
_보조 강조_
~~취소선~~

> 주의사항이나 핵심 메시지를 강조할 때 사용합니다.
```

강조가 많으면 아무것도 강조되지 않습니다. 한 문단에 굵은 글자가 너무 많아지지 않게 합니다.

## GitHub README 꾸미기

Markdown만으로는 표현이 제한적입니다. GitHub README에서는 일부 HTML과 배지 이미지를 활용할 수 있습니다.

### 중앙 정렬

```md
<h1 align="center">Frontend Guide</h1>

<p align="center">
  프론트엔드 실무 가이드
</p>
```

### 배지

```md
![Git](https://img.shields.io/badge/Git-workflow-f05032?style=flat-square&logo=git&logoColor=white)
```

### 배너 이미지

외부 이미지 서비스를 사용할 수 있지만, 저장소가 오래 유지되어야 한다면 직접 만든 이미지를 `assets/`에 넣는 방식이 가장 안정적입니다.

```md
<p align="center">
  <img src="./assets/banner.png" alt="Frontend Guide banner">
</p>
```

꾸밈은 첫 화면을 보기 좋게 만들 수 있지만, 본문 가독성을 해치면 안 됩니다. 문서의 핵심은 링크와 구조입니다.
